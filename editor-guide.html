Часть I. Основы

1. Введение: что вы создаёте

JSON RPG Editor — визуальный конструктор текстовой RPG на базе D&D 5e с опциональной поддержкой Pathfinder 2e (отдельные классы и родословные в том же файле данных). Вся игра — один game_data.json, в котором описаны:

  • Сцены — экраны с текстом, диалогами, кнопками и опционально боем.
  • Выборы — куда ведёт игрок, проверки навыков, скрытые ветки по условиям.
  • Расы и классы — создатель персонажа: сначала раса, затем класс; бонусы расы (ASI или PF2e Boosts), подсказки из ui_hints.
  • Умения и прогрессия — бой, ресурсы, уровни 1–10, ASI, концентрация, ячейки заклинаний.
  • Флаги и квесты — память сюжета (flags) и журнал со стадиями (questStages, questSet).
  • Репутация и торговля — числовые флаги rep_*, скидки в лавке.
  • Расходники — лечение, урон по врагам, свитки с useAbility; в бою — выпадающий список предметов.
  • Добыча с врагов — таблица loot, окно награды после победы.
  • Компонентные сцены — лавка, кузница, храм, диалог собираются из блоков в редакторе (§29.1).
  • Цепочки действий — покупка, лечение, квесты без кода: атомарные шаги в конструкторе (§29.2).
  • Карта мира — быстрые переходы между открытыми локациями.

Редактор (editor.html) меняет JSON. Игра (index.html) читает те же данные и запускает сюжет. Вам не нужно писать код, если вы используете встроенные механики: условия, квесты, бой, предметы, проверки навыков.

Новое в движке

2. Обновления движка (после первой версии руководства)

Краткий список механик, уже работающих в index.html и описанных ниже подробнее.

  • Визуальный конструктор условий — правила showIf/hideIf/states без ручного JSON (вкладка сцен, превью условий).
  • Квесты по стадиям — questSet на выборе, журнал «Текущие задания», правило «Стадия квеста» в условиях.
  • Ячейки заклинаний — у мага и паладина (с 2 уровня) ресурс отображается точками по кругам; у воина — «ярость» (очки энергии).
  • Расходники с умением — у предмета поле useAbility (ID из progression.abilities), например свиток огненного шара.
  • Репутация — числовые флаги rep_village и др.; влияние на цены; статусы «Вражда / Нейтралитет / Дружба / Герой» в сайдбаре; условия min / max на флаг.
  • Концентрация (D&D 5e) — одно длительное заклинание; проверка Телосложения при уроне; индикатор [C] в бою.
  • Зелье фокусировки — преимущество на проверки концентрации 1 час (лавка Джека, 22 зм).
  • Прогрессия до 10 уровня — при повышении выбор 1 умения из 2–3 вариантов; на 4, 8 и 10 уровнях — ASI (+2 к одной характеристике или +1 к двум).
  • Создатель персонажа — раса → класс → характеристики (Point Buy или PF2e Boosts) → навыки → бонусное умение 1 уровня.
  • Расы (D&D 5e) — вкладка 🧬 Расы: ASI, черты, языки; бонусы применяются после Point Buy (потолок 20).
  • Родословные (PF2e) — расы с system: "pf2e", фиксированные и свободные Boosts, обязательное наследие (heritages), ОЗ от расы.
  • Классы D&D — воин, маг, паладин, плут, жрец, следопыт; классы PF2e — pf2e_* с полем system: "pf2e".
  • Подсказки UI — блок ui_hints для классов и характеристик в создателе персонажа.
  • Дополнительные квесты — в демо-модуле: «Тень Люкорна» (lukorn_investigation), «Медальон Эльзы» (albert_locket).
  • Plugin API для special — SpecialSceneRegistry, JSON в plugins.specialScenes, свои обработчики в отдельных JS без правки engine.js.
  • Аксессуары — кольца (2 слота), ожерелье, серьги; бонусы в bonuses.
  • Расходники с целью — use.target: себя, один враг, все враги; выбор цели в бою.
  • Добыча (loot) — у врага таблица выпадения; после боя — модальное окно «Добыча».
  • Компонентные сцены — сцена = контейнер: диалог, торговля, ремонт, заточка, лечение и кнопки-цепочки (§29).
  • Цепочки действий — actionChains в JSON, вкладка 🔗 Действия, переиспользуемые сценарии (покупка зелья, лечение в храме).
  • Заточка и проклятия — компоненты upgrade / curse_remove или цепочки; уровень заточки в сохранении.
  • Проклятые предметы — секция «Проклятие» во вкладке Предметы; снятие через компонент храма (§25.3).
  • Иерархия мира — мир → регион → хаб → сцена; фон/музыка наследуются, компоненты — нет (вкладка 🌍 Мир).
  • Шаблоны сцен — вкладка 📋 Шаблоны, создание локации из типа (лавка, кузница…).
  • Навыки D&D 5e — у класса skillChoices (сколько выбрать и из какого списка); отдельный шаг в создателе; владения в state.proficiencies.skills; бонус мастерства в проверках (§19.4, §39).
  • Навыки PF2e — fixedSkills + выбор из пула с рангом trained; ранги T/E/M/L в state.skills; Skill Increase на чётных уровнях (§19.5, §24, §39).
  • Dock в игре — иконки 🎒 ⚡ 📍 🔊 📁 справа внизу (на телефоне — снизу); панели поверх сюжета, не сдвигают текст (§34.2).
Патч js/patch-progression.js при загрузке данных подставляет таблицы уровней 2–10, ASI и ячейки — правки в JSON можно дублировать в редакторе или оставить патчу. После правок game_data.json выполните node scripts/sync-data-js.js, чтобы обновить js/data.js.

Как данные попадают в игру

  1. Вы редактируете проект в editor.html.
  2. Нажимаете Сохранить JSON — скачивается game_data.json.
  3. Для теста: положите файл в data/ и выполните node scripts/sync-data-js.js или в игре нажмите Загрузить контент JSON и выберите файл.
  4. Для игроков: Экспорт игры — папка с index.html и уже вшитым js/data.js.

4. Запуск редактора и игры

Редактор

  1. Откройте editor.html в Chrome или Edge.
  2. Для кнопки Экспорт игры (запись папки на диск) удобен локальный сервер, например Live Server в VS Code — адрес вида http://127.0.0.1:5500/..., не file://.
  3. Нажмите Загрузить JSON и выберите data/game_data.json или свой экспорт.

Игра

  1. Откройте index.html.
  2. После правок в редакторе обновите данные (см. §3) или загрузите JSON вручную в игре.
  3. Сохранение прогресса игрока — кнопки в сайдбаре игры (браузерное хранилище).

Часть II. Создание игры

6. Создание игры с нуля: пошаговый план

  1. Новый проект — введите название игры.
  2. Мета — при PF2e-кампании укажите meta.system: "pf2e" (§11).
  3. Расы — хотя бы одна раса под выбранную систему (§20).
  4. Классы — минимум один класс с тем же system, что у проекта (§19).
  5. Предметы — оружие, зелья, ключи (ID понадобятся в наградах и условиях).
  6. Враги — если планируется бой.
  7. Сцены — цепочка от start: текст + выборы «Переход».
  8. Стартовый экран — сцены выбора класса с action: select_class:ИД и сцена имени с start_game.
  9. Квесты — создайте квест, стадии 0, 1, 2…; на выборах укажите questSet для смены стадии.
  10. Условия — конструктор «Показать если» на кнопках (флаги, предметы, стадия квеста).
  11. Карта — точки и mapLocation на сценах.
  12. Тема и звуки — по желанию.
  13. Проверить проект → исправить замечания.
  14. Экспорт игры или Сохранить JSON → пройти игру в index.html с нуля.

7. Практикум: мини-кампания «Пропавшая реликвия» (пошагово)

Ниже — полный цикл без кода: от пустого проекта до играбельной ветки с квестом, условием и наградой. Повторяйте действия в редакторе параллельно с чтением.

Шаг A. Подготовка данных

  1. Новый проект → название «Реликвия».
  2. Классы → класс hero, название «Герой», ОЗ 20, КД 14, ресурс «Энергия» max 2.
  3. Предметы → old_key (misc), relic (misc), heal_potion (consumable).
  4. Квесты → ID relic_quest, стадия 0: log «Священник просит вернуть реликвию», hint «Поговорите в храме»; стадия 1: hint «Найдите реликвию в склепе»; стадия 2: finish ✓, log «Реликвия возвращена».

Шаг B. Сцены сюжета

  1. Сцена class_pick — текст «Выберите путь», выбор select_class:hero.
  2. Сцена name_entry — кнопка start_game → temple (после создателя персонажа в игре).
  3. Сцена temple — текст «Священник ждёт у алтаря».
    • Выбор «Принять поручение» → crypt_entrance, блок questSet: квест relic_quest, стадия 0.
    • Выбор «Уйти» → village (без квеста).
  4. Сцена crypt_entrance — showIf: стадия квеста relic_quest = 0 или 1 (режим «Любое» с двумя правилами questStage, либо одна стадия 1 после принятия).
    • Выбор «В склеп» → crypt_inside только если нет предмета relic (правило «Нет предмета»).
  5. Сцена crypt_inside — skillCheck Восприятие DC 12: успех → флаг found_niche, предмет old_key; провал → остаётесь без ключа.
  6. Сцена relic_room — showIf: флаг found_niche = true И hasItem old_key. Выбор «Взять реликвию» → temple, questSet стадия 1, награда предмет relic на сцене (items).
  7. Сцена temple — второй выбор «Сдать реликвию» — showIf: hasItem relic + questStage 1; questSet стадия 2; золото 50.

Шаг C. Проверка

  1. На сцене temple откройте Превью видимости: без реликвии кнопка «Сдать» скрыта; с реликвией и стадией 1 — видна.
  2. Проверить проект → исправьте битые to.
  3. Карта сюжета — убедитесь, что нет изолированных узлов.
  4. Сохраните JSON → откройте игру → пройдите путь от класса до сдачи реликвии.
Этот практикум использует только формы редактора: квесты, questSet, конструктор условий, skillCheck. JSON понадобится только для consumable use, если хотите лечащее зелье.

9. Вкладка JSON: когда она нужна

Вкладка JSON показывает весь проект. Для сюжета она не обязательна. Используйте её, когда:

  • Нужно скопировать фрагмент сцены или квеста между проектами.
  • Поле ещё не вынесено в форму: repeatRewards, use у зелья, content у книги, ui_hints, statusEffects.
  • Ищете опечатку в ID по всему файлу (Ctrl+F).
После ручного редактирования JSON не нажимайте «Сохранить» в браузере на вкладке — данные живут в памяти редактора. Ошибка синтаксиса (лишняя запятая) может сломать экспорт. Проверяйте валидность и снова Загрузите JSON, если что-то пошло не так.

9.1 Диалоги на сцене

Кнопка + Добавить в секции диалогов. Каждая строка: Говорящий + Текст. В игре реплики идут после основного абзаца, отдельными блоками.

Пример диалога 
"dialogue": [
  { "speaker": "Марта", "text": "Опять чужаки… Чего вам?" },
  { "speaker": "Вы", "text": "Ищу дорогу к мельнице." }
]

Имя говорящего — произвольная строка; связь со вкладкой NPC не обязательна, но удобно писать те же имена, что в справочнике NPC.

10. Метаданные проекта

Слева внизу блок Проект → ✏️ Мета. Заполняются:

  • meta.title — название (заголовок окна при экспорте).
  • meta.author — автор.
  • meta.description — краткое описание.
  • meta.version — версия (при новом проекте «1.0»).
  • meta.system — активная система правил для создателя и боя (см. §11). Если поле пустое, движок выводит систему из содержимого JSON (наличие PF2e-классов/рас).

11. Системы правил (D&D 5e / Pathfinder 2e)

Один проект может содержать оба набора классов и рас, но в игре активна одна система — из meta.system:

Значение meta.systemПоведение
dnd5e (по умолчанию)Point Buy 27 очков, владения навыков (skillChoices), бонус мастерства в проверках, расы без system: "pf2e".
pf2eAbility Boosts, ранги навыков (T/E/M/L), Skill Increase на чётных уровнях, наследие обязательно, классы/расы с system: "pf2e".
genericУпрощённые правила, если в проекте нет заклинаний (редко для готовых модулей).

11.1 Как разделить контент в JSON

  • Классы PF2e — ID вида pf2e_fighter, поле "system": "pf2e" в объекте класса.
  • Родословные PF2e — ID вида human_pf2e, поле "system": "pf2e" в объекте расы.
  • Классы и расы D&D — поле system отсутствует или не равно pf2e.
Не меняйте ID базовых классов демо (warrior, wizard, paladin) — на них завязаны сцены select_class. Новые классы D&D добавляйте отдельными ID (rogue, cleric, ranger).

11.2 Новый проект и шаблон PF2e

При создании проекта движок и редактор подставляют meta.system по содержимому. Для чистой PF2e-кампании задайте meta.system: "pf2e" в метаданных или используйте шаблон PF2e в редакторе (если доступен в вашей сборке).

12. Сцены: полный разбор полей

Чтобы выдавать снова (фарм), во вкладке JSON у сцены добавьте: "repeatRewards": true.

12.5 Компонентные сцены (кратко)

Если у сцены задан массив components, в игре под текстом появляется панель блоков (лавка, кузнец, кнопки), а не только список выборов. Настройка — в форме сцены, секция 🧩 Компоненты; подробно — §29.

Демо: blacksmith (диалог + ремонт + заточка + торговля), jack_shop (приветствие + лавка + быстрая покупка зелья), temple (диалог + лечение + снятие проклятий).

13. Состояния локации (states)

Одна и та же сцена может выглядеть по-разному в зависимости от прогресса — без дублирования десятка сцен. Блок 🔄 Состояния локации появляется под текстом сцены.

Как это работает

  1. Движок проверяет states[] сверху вниз.
  2. Для каждого элемента вычисляется condition (визуальный конструктор — как у showIf).
  3. Первое истинное состояние подменяет: text, location, audio, choices, special.
  4. Если ни одно не подошло — используется default (галочка Fallback) или базовые поля сцены.
  5. Бой, флаги сцены, предметы при входе не меняются states — только отображение.
Пример: деревня до и после пожара 
"states": [
  {
    "id": "burning",
    "condition": {
      "all": [
        { "flag": "village_on_fire", "equals": true }
      ]
    },
    "text": "Деревня охвачена пламенем. Жители бегут к реке.",
    "choices": [
      { "text": "Помочь тушить", "to": "fight_fire", "icon": "🔥" }
    ]
  },
  {
    "id": "peaceful",
    "default": true,
    "text": "Тихая площадь. Пахнет свежим хлебом."
  }
]
  1. Откройте сцену → + Добавить в блоке состояний.
  2. Задайте ID состояния и Условие (condition) в конструкторе (например флаг = true).
  3. Заполните override-текст и при необходимости свои выборы.
  4. У одного состояния отметьте Fallback (default) — оно покажется, если другие условия ложны.

16. Смена стадии квеста на выборе (questSet)

Чтобы продвинуть квест без ручного флага, в карточке выбора есть блок 📜 При выборе — стадия квеста.

  1. Нажмите + При смене стадии квеста.
  2. Выберите квест и стадию из списков (стадии из вкладки Квесты).
  3. Игрок при нажатии этой кнопки получит новую стадию в журнале.
Пример: взять квест у NPC 
{
  "text": "Я помогу найти мельника",
  "to": "village_after_quest",
  "questSet": { "questId": "find_albert", "stage": "0" }
}
Пример: завершить квест (стадия с finish: true) 
{
  "text": "Сдать отчёт",
  "to": "reward_scene",
  "questSet": { "questId": "find_albert", "stage": "3" }
}
Связывайте видимость других кнопок с правилом Стадия квеста в showIf, а не с отдельными флагами quest_* — так проще сопровождать сюжет.

На выборе можно задать изменение репутации (в JSON вкладки сцены):

"reputation": { "rep_village": 5 }

При нажатии кнопки движок вызовет changeReputation и запишет в журнал при заметном изменении.

17.4 Репутация (rep_*)

В демо-модуле «Мельница на Тихой реке»:

  • Стартовое значение: startingFlags.rep_village: 0 в game_data.json.
  • Метаданные фракции: reputation.rep_village (название «Деревня», иконка).
  • Блок Отношения в сайдбаре — только если флаг уже есть в сохранении; статусы без цифр: Вражда (<−10), Нейтралитет, Дружба, Герой (≥25).
  • Лавка Джека / компонент trade: цена × модификатор репутации; при rep_village ≤ −20 — отказ в торговле (блок лавки покажет отказ).

В редакторе для условий числового флага пока удобнее вкладка JSON на выборе: "min": 10 или "max": -20 рядом с flag.

18. Квестовая система

Квесты живут во вкладке 📜 Квесты. В игре отображаются в Журнале заданий по полям log (текст записи) и hint (подсказка «что делать»).

18.1 Создание квеста

  1. + Новый квест → введите ID (find_albert).
  2. Заполните название и описание.
  3. + Стадия — стадии нумеруются 0, 1, 2… (в JSON ключи строки "0", "1").
  4. У каждой стадии: log, hint, галочка Завершение квеста (finish) на финальной стадии.
  5. Опционально: награда XP, скрытый квест (hidden).
Пример квеста в JSON (после сохранения из редактора)

  "hidden": false,
  "rewards": { "exp": 150, "gold": 25 },
  "stages": {
    "0": {
      "log": "Марта просит найти мельника Альберта.",
      "hint": "Поговорите с трактирщицей и осмотрите мельницу.",
      "finish": false
    },
    "1": {
      "log": "Вы нашли следы у амбара.",
      "hint": "Осмотрите амбар и погреб.",
      "finish": false
    },
    "3": {
      "log": "Альберт спасён. Деревня благодарна.",
      "hint": "",
      "finish": true
    }

21. Создатель персонажа в игре

После select_class (класс уже выбран на сцене) открывается пошаговый мастер character-creator.js.

21.1 D&D 5e

  1. Раса — сетка доступных рас; отображаются бонусы ASI к характеристикам.
  2. Класс — подтверждение или смена (только классы текущей системы).
  3. Point Buy — 27 очков, характеристики 8–15; затем применяется ASI расы (макс. 20).
  4. Навыки — отдельный шаг: выбор из skillChoices (до count штук) + автоматически bonusSkills расы (не входят в лимит).
  5. Умение 1 уровня — одно из level1BonusChoices.
  6. Имя и финализация.

21.2 Pathfinder 2e

  1. Раса и наследие (heritage) — обязательны.
  2. Класс — только pf2e_*.
  3. Ability Boosts — фиксированные и свободные бусты вместо Point Buy (потолок 18 на 1 уровне).
  4. Навыки (PF2e) — сначала fixedSkills класса, затем выбор из skillChoices; каждому навыку присваивается ранг (обычно trained).
  5. Умение 1 уровня из level1BonusChoices, имя, финализация.

21.3 Подсказки (ui_hints)

В JSON можно задать тексты для панелей при наведении / выборе:

  • class_warrior, class_wizard, class_paladin, class_rogue, class_cleric, class_ranger — описание классов.
  • stat_str … stat_cha — пояснения характеристик.
  • Также: hp, ac, inventory, travel и др. для сайдбара игры.

Редактируются во вкладке JSON или при копировании из демо-модуля.

21.4 Умения и ОЗ при создании

  • Базовые умения — из classes.*.abilities (воин: Второе дыхание, Всплеск, Блок щитом; маг: снаряд, щит, пылающие руки…).
  • Дополнительное умение — одно из level1BonusChoices (патч patch-progression.js или JSON класса).
  • Активные черты расы (traits type active) могут попасть в список умений.
  • ОЗ: кость класса + мод. ТЕЛ (D&D) или логика PF2e с races.*.hp.

21.5 Навыки в игре и проверки

В сайдбаре блок Владения (D&D) или список навыков с рангом и итоговым бонусом (PF2e), например «Атлетика (T +5)».

  • При проверке навыка на выборе сцены (§14.2) движок бросает 1d20 + бонус и сравнивает с DC.
  • D&D: зелёная ✓ на кнопке, если навык в state.proficiencies.skills.
  • PF2e: цветная метка ранга (U/T/E/M/L) на кнопке проверки.
  • На чётных уровнях PF2e после ASI/умения может открыться окно Увеличение навыка — повысить один навык на одну ступень ранга (§24).

24. Прогрессия и уровни (1–10, ASI)

Вкладка 📈 Прогрессия и патч js/patch-progression.js задают поведение в игре.

  • maxLevel: 10, опыт: 0, 80, 180, 320, 500, 750, 1100, 1550, 2100, 2800 (пороги уровней).
  • На уровнях 2–10 у каждого класса в progression.levels — массив choices: 2–3 ID умений; игрок обязан выбрать одно в модальном окне.
  • На уровнях 4, 8 и 10 — флаг asi: true: сначала окно ASI (+2 к одной характеристике или +1 к двум), затем выбор умения (если есть choices).
  • Ячейки заклинаний мага/паладина и ярость воина обновляются по таблицам в патче при каждом уровне.
  • PF2e — Skill Increase: на чётных уровнях (2, 4, 6, 8, 10) после повышения уровня игрок выбирает один навык и повышает его ранг (trained → expert → master → legendary). История — в state.skillIncreases.
Пример: уровень 4 воина в JSON класса 
"4": {
  "asi": true,
  "choices": ["warrior_armor_mastery", "death_strike"]
}
Старые сохранения (например, 3 уровень) не теряют умения; новые выборы появятся только при следующем повышении уровня.

25. Предметы и экипировка

  1. Вкладка 🎒 Предметы → слева список (☠️ у проклятых) → справа карточка.
  2. + Новый предмет → ID латиницей.
  3. Тип: weapon, armor, shield, accessory, consumable, readable, misc, key, quest.
  4. Для оружия/брони/щита: урон или КД, блок Бонусы (JSON), опционально Заточка.
  5. Для расходника — блок Использование (use) (см. §25.2).
  6. Секция ☠️ Проклятие — только через форму (§25.3).

В игре: инвентарь в dock 🎒 → Надеть/Снять. КД, урон, ОЗ и характеристики пересчитываются с учётом всех слотов и заточки. Проклятые надетые вещи нельзя снять вручную — только у священника (§25.3).

У предмета в JSON можно задать price — базовая цена для компонента trade и legacy-лавки special: shop.

Умение scroll_fireball должно существовать в progression.abilities. В бою — кнопка «Использовать»; не тратит ячейки класса; предмет сгорает.

Пример: зелье фокусировки 
"focus_potion": {
  "name": "Зелье фокусировки",
  "type": "consumable",
  "use": { "effect": "focus_potion", "durationHours": 1 }
}

Преимущество на проверки удержания концентрации 1 час; продаётся у Джека (22 зм).

25.1. Аксессуары (кольца, ожерелье, серьги)

Тип предмета accessory. В редакторе:

  • Слот: ring (кольцо 1 / кольцо 2), necklace, earrings.
  • Бонусы (JSON) — те же поля, что у оружия: maxHpBonus, acBonus, atkBonus, dmgBonus, str…cha.
  • Иконка (emoji) — отображается в инвентаре и слотах.

В игре четыре слота аксессуаров в панели экипировки (🎒). Два кольца — разные слоты ring1 и ring2.

Пример: кольцо жизни 
"ring_of_vitality": {
  "name": "Кольцо жизни",
  "type": "accessory",
  "slot": "ring",
  "icon": "💍",
  "bonuses": { "maxHpBonus": 10, "con": 1 }
}

25.2. Расходники: цель использования и бой

У типа consumable в редакторе появляется блок Использование (use):

ПолеЗначение
effectheal, damage, focus_potion, rest_material, message
targetself — себя; ally — союзник (пока как self); single_enemy — один враг в бою; all_enemies — все живые враги
formula / amountКубики урона/лечения, например 2d4+2
labelТекст кнопки («Выпить», «Бросить»…)

Вне боя: кнопка в инвентаре (🎒). Для single_enemy без боя предмет не сработает.

В бою: под кнопками атаки — выпадающий список «🎒 Предмет» и Использовать; для цели «один враг» — фаза выбора цели (как у умения), кнопка «Отмена» / Escape.

Пример: бомба по одному врагу 
"throwable_bomb": {
  "name": "Бомба",
  "type": "consumable",
  "icon": "💣",
  "use": {
    "effect": "damage",
    "formula": "2d6",
    "target": "single_enemy",
    "label": "Бросить"
  }
}

25.3. Проклятые предметы

Настраивается только в редакторе (секция «Проклятие» на карточке предмета), без ручного JSON.

  1. Включить чекбокс Проклятый предмет.
  2. Эффекты — выбрать из списка и «Добавить эффект»: silence, weakness, poison_touch, bloodlust, haunted (минимум один).
  3. Стоимость снятия (зм) — плата священнику.
  4. Сцена снятия — ID сцены с компонентом curse_remove (в демо: temple) или legacy special: temple_priest.

Поля в JSON (если смотрите вкладку JSON): cursed, curseEffects[], curseRemoveCost, curseRemoveScene. Старые предметы без полей считаются обычными.

В игре:

  • При надевании — сообщение в журнале, эффекты в панели умений (⚡). Например silence блокирует заклинания с spellLevel ≥ 1.
  • Снять через инвентарь нельзя — кнопка «⚠️ Проклято».
  • В храме компонент ✨ Снятие проклятий (или legacy-сцена temple_priest) снимает проклятие с надетых вещей за золото; предмет попадает в инвентарь, но остаётся cursed: true — при повторном надевании эффекты вернутся.
  • Проклятое нельзя продать в лавке.
Пример + тестовая сцена 
"silent_knight_armor": {
  "name": "Латы безмолвного рыцаря",
  "type": "armor",
  "ac": 18,
  "bonuses": { "maxHpBonus": 15, "atkBonus": 2 },
  "cursed": true,
  "curseEffects": ["silence"],
  "curseRemoveCost": 50,
  "curseRemoveScene": "temple_priest"
}

Рекомендуемая сцена: temple с компонентом curse_remove. Legacy: отдельная сцена temple_priest. Тест: dev_cursed_armor (выдаёт латы и золото).

25.4. Заточка (кузница)

На карточке оружия/брони/щита — блок ⚒️ Заточка:

ПолеСмысл
enhancementСтартовый уровень в шаблоне (обычно 0)
enhancementMaxМаксимум (+N), по умолчанию 3
enhancementCostЦены за уровни через запятую, напр. 100, 300, 900

Фактический уровень заточки хранится в сохранении (itemEnhancements), не в JSON проекта — у каждого игрока свой прогресс.

Сцена кузницы: в демо — blacksmith с компонентами upgrade (заточка) и repair (сброс уровня заточки за плату). Legacy: special: "blacksmith" на сценах без components.

Игрок видит надетое оружие/броню/щит и кнопки «Закалить за N зм». Усиливает atk/dmg/ac по правилам движка (+1 за уровень заточки). Тест: dev_blacksmith, предмет test_sword.

26. Бой: концентрация, ячейки, интерфейс

26.1 Ячейки заклинаний и ярость

В сайдбаре под ресурсом класса:

  • Маг — точки по кругам (1–5), тратятся при spellLevel умения.
  • Паладин — полукастер: ячейки с 2 уровня.
  • Воин — «Ярость» (очки энергии), тратятся по полю cost.

Восстанавливаются при отдыхе (полностью с припасами, частично без).

26.2 Концентрация

Умения с concentration: true (щит веры, удержание личности, зеркальный образ…):

  • Только одно активное заклинание; новое снимает старое.
  • При получении урона — спасброс Телосложения (DC 10 или половина урона).
  • В бою: метка [C] Название; при провале — «Концентрация потеряна!»
  • Концентрация сбрасывается в конце боя.

26.3 Статус-эффекты

Яд, кровотечение, оглушение, реген — вкладка данных statusEffects и эффект apply_status в умениях.

26.4. Интерфейс боя (обновление)

Центральная зона боя показывает:

  • Полоски ОЗ врагов, кнопки Атаковать [имя].
  • Умения класса (если хватает ресурса/ячейки и нет блокировки проклятием).
  • Строка 🎒 Предмет — расходники с use, пригодные в бою (§25.2).
  • Для PF2e — индикатор оставшихся действий (◆).
  • Концентрация [C], статусы на врагах.

Умения с выбором цели (один враг) и расходники single_enemy переводят бой в фазу выбора цели; подсказка над выборами, «Отмена» сбрасывает фазу.

Удобный доступ к умениям вне длинного сайдбара — dock ⚡ (панель шире ~400px, описание переносится).

26.5. Добыча после победы

После уничтожения всех врагов движок бросает таблицы loot каждого участника боя, затем показывает модальное окно «Добыча». Игрок нажимает Забрать — предметы и золото попадают в инвентарь/кошелёк, затем переход на nextScene сцены.

Награды сцены (items, gold на самой сцене) по-прежнему выдаются при входе на nextScene, не в окне добычи.

Настройка таблицы — §27.3.

27. Враги, бой, масштабирование

27.1 Создание врага

Вкладка Враги — слева список всех врагов, справа карточка выбранного: имя, ОЗ, КД, бонус атаки, кубик урона, ловкость (инициатива), опыт. Чекбокс Босс — усиленное масштабирование ОЗ. Внизу списка — + Добавить врага.

27.2 Масштабирование по уровню игрока

Вкладка Прогрессия → блок Масштабирование врагов (enemyScaling):

  • ОЗ за уровень — множитель для обычных врагов.
  • ОЗ за уровень (босс) — для врагов с boss: true.
  • Бонус атаки за чётный уровень игрока.
  • Таблица +КД — на 5, 10 уровне и т.д.

На 1 уровне игрока враги без усиления; дальше параметры растут автоматически в бою.

27.3. Таблица добычи (loot)

На карточке врага (вкладка Враги) — секция Добыча (loot):

КолонкаОписание
ПредметID из items или gold для монет
Шанс0–1 (например 0.5 = 50%)
min / maxСколько штук (или зм) при успешном броске

+ Добавить строку / ❌ удалить. Каждая строка бросается отдельно; одинаковые предметы суммируются в окне добычи.

Пример: крыса с хвостом и золотом 
"rat": {
  "name": "Гигантская крыса",
  "hp": 7,
  "loot": [
    { "item": "rat_tail", "chance": 0.6, "min": 1, "max": 1 },
    { "item": "gold", "chance": 0.3, "min": 2, "max": 8 }
  ]
}

Боевая сцена должна перечислить этого врага в combat. После победы — модальное окно (§26.5).

28. Справочник NPC

Вкладка NPC — слева список персонажей, справа карточка (имя, иконка, описание, локация, отношение). + Добавить NPC внизу списка. Движок не подставляет NPC в сцены автоматически: реплики пишите в диалогах сцены. Справочник помогает держать единые имена и описания.

29. Компонентные сцены и цепочки действий

Современный способ собрать лавку, кузницу, храм и диалоги — без поля special. Сцена остаётся обычной (текст, награды, выход), а интерактив рисуется из блоков components и переиспользуемых actionChains.

Правило: если у сцены есть непустой components, движок не вызывает special. Старые сцены только с special: shop / blacksmith продолжают работать как раньше (§29.4).

29.1 Компонентные сцены

В редакторе сцены, секция 🧩 Компоненты сцены: добавляйте блоки, меняйте порядок (↑↓), отключайте чекбоксом «Вкл», смотрите Предпросмотр.

В игре блоки показываются в зоне #scene-components-area под текстом, сверху вниз. Кнопка 🚪 Уйти — из exitScene / хаба (как у деревенских локаций).

ТипНазначение
💬 dialogueПриветствие NPC, темы разговора; опционально chainOnEnter — цепочка при входе.
💰 tradeПолная лавка: покупка/продажа, ассортимент из shopInventories, репутация.
⚒️ repair«Ремонт» — сброс уровня заточки надетых вещей за золото.
⬆️ upgradeЗаточка экипированного оружия/брони/щита (maxEnhancement, costTable).
🩹 healПлатное лечение ОЗ и опционально ресурса класса.
✨ curse_removeСнятие проклятия с надетых предметов (§25.3).
🎲 gamblingПростая игра на ставку (кости).
🔨 craftingКрафт по списку рецептов в params.recipes.
🔘 interactiveОдна кнопка → запуск цепочки params.chain (§29.2).
Формат компонента в JSON 
{
  "component": "trade",
  "id": "jack_trade",
  "enabled": true,
  "params": {
    "merchant": "jack",
    "inventory": "village_shop",
    "sellMultiplier": 1,
    "buyMultiplier": 0.5,
    "repFaction": "rep_village"
  },
  "conditions": {
    "showIf": { "flag": "jack_met", "value": true },
    "hideIf": { "flag": "jack_dead", "value": true }
  }
}

Условия showIf / hideIf — те же правила, что у выборов (§15). Компоненты не наследуются от хаба: только фон и музыка (§ вкладка «Мир»).

29.2 Цепочки действий (actionChains)

Цепочка — упорядоченный список атомарных шагов: проверить золото → забрать → выдать предмет → сообщение в лог. Одну цепочку можно повесить на кнопку в разных сценах (храм, лагерь, лавка).

Вкладка «🔗 Действия»

  1. Слева — список цепочек (buy_potion, heal_at_temple…).
  2. Справа — шаги: добавление по категориям (инвентарь, экономика, здоровье…), ↑↓, drag-and-drop, параметры каждого шага.
  3. Для проверок (золото, предмет, флаг) — поле При неудаче: ID другой цепочки (например not_enough_gold).
  4. Тестовый запуск — симуляция на тестовом герое без входа в игру.
  5. 💾 Сохранить — данные попадают в actionChains в JSON; не забудьте node scripts/sync-data-js.js.

Пример: покупка зелья

"buy_potion": {
  "name": "Купить зелье лечения",
  "steps": [
    { "action": "check_gold", "params": { "amount": 20 }, "onFail": "not_enough_gold" },
    { "action": "remove_gold", "params": { "amount": 20 } },
    { "action": "add_item", "params": { "itemId": "healing_potion", "count": 1 } },
    { "action": "log", "params": { "message": "✅ Куплено зелье!", "type": "success" } },
    { "action": "refresh_ui" }
  ]
}

Ветвление (if / else)

У шага с returns: boolean (или у check_gold, check_item, check_flag, check_skill, roll_check) можно задать:

  • onFail — строка (ID цепочки) или массив шагов;
  • onSuccess — массив шагов при успехе (основная цепочка после такого шага не продолжается, если сработала ветка).

Реестр действий (для справки)

Категории: 📦 инвентарь (add_item, remove_item, check_item), 💰 экономика, ❤️ здоровье (heal, damage, эффекты), 🎭 сцены (change_scene, set_flag, update_quest), 🗣️ диалог (say, show_choices), ⚔️ бой, 🔮 баффы, 🧩 утилиты (log, roll_check, random).

Выборы сцены могут вызывать цепочку: "action": "chain:buy_potion".

29.3 Торговля, кузница, храм (практика)

Лавка Джека (jack_shop)

  1. Компонент dialogue + chainOnEnter: jack_greeting.
  2. Компонент trade — ассортимент village_shop, NPC jack.
  3. Компонент interactive — «Купить зелье (20 зм)» → цепочка buy_potion.
  4. Ещё кнопки: sell_item, leave_shop (переход на village_hub).

Кузница (blacksmith)

Компоненты: диалог с кузнецом, repair, upgrade (цены costTable), trade с ассортиментом smithy_ore. Ассортименты задаются в корне JSON: shopInventories.

Храм (temple)

Диалог (тема «Пожертвовать» через donate в JSON темы), компоненты heal, curse_remove, кнопка на цепочку heal_at_temple (50 зм, формула 2d8+4) — ту же цепочку можно использовать в лагере.

29.4 Legacy: поле special

Для старых сцен и сюжета демо по-прежнему работает SpecialSceneRegistry (js/special-scenes.js). Пока у сцены нет массива components, при special вызывается встроенный обработчик.

  • Клик по узлу — открыть сцену в редакторе.

34. Интерфейс для игрока

34.1. Сайдбар (слева)

Компактная колонка без длинной прокрутки:

  • Персонаж — имя, раса, класс, ОЗ, КД, атака, урон, золото, уровень/опыт.
  • Ресурс — ярость или ячейки заклинаний (точки по кругам).
  • Характеристики; блок Владения / Навыки — D&D: список владений с бонусом мастерства; PF2e: навык, ранг (T/E/M/L) и итог (+N); блок Отношения (репутация).
  • Текущие задания — журнал квестов.
  • Отдых — припасы и кнопка «Отдохнуть».

34.2. Dock справа внизу

Вертикальная полоска иконок; панели открываются поверх сюжета (контент не сдвигается). Одновременно открыта одна панель; повторный клик по иконке или ❌ — закрыть.

ИконкаПанель
🎒Инвентарь — экипировка, слоты, предметы, надеть/снять
⚡Умения — список с «Использовать», активные проклятия ☠️
📍Локация — название места, меню «Переместиться…»
🔊Звук — громкость музыки/эффектов, вкл/выкл
📁Меню — загрузка JSON, сохранение, сброс, выход к выбору игры

На узком экране (<768px) dock внизу горизонтально, панели выезжают снизу.

34.3. Центр экрана

Текст сцены, диалоги, при локациях с компонентами — блоки лавки/кузницы/кнопок (§29), зона боя (§26.4), кнопки выборов и «Уйти». После победы — окно Добыча, затем переход по сцене.

Повышение уровня: модальное окно — ASI (4/8/10) и/или выбор одного умения; для PF2e на чётных уровнях — также увеличение навыка (§24).

Журнал: броски, урон, квесты, репутация, концентрация.

  1. Есть цепочка от выбора класса до первой игровой сцены.
  2. У каждого класса есть select_class:ИД; расы и классы соответствуют meta.system.
  3. У каждого класса настроены навыки: D&D — skillChoices; PF2e — fixedSkills + пул.
  4. skillCheck в сценах используют ID из §39 для активной системы правил.
  5. В races есть хотя бы одна раса; у PF2e — наследия в каждой родословной.
  6. Все to, successNext, failNext, nextScene существуют.
  7. Бои имеют nextScene.
  8. Квесты продвигаются через questSet; финальная стадия с finish.
  9. Репутация: startingFlags, блок reputation, ветки по min/max.
  10. Умения в пуле совпадают с ID в progression.levels.*.choices; расходники с useAbility есть в пуле.
  11. Условия на ключевых ветках проверены в превью.
  12. Локации-сервисы: jack_shop, blacksmith, temple — компоненты и цепочки работают; при необходимости legacy special не конфликтует с components.
  13. Цепочки actionChains проверены тестовым запуском во вкладке «Действия».
  14. Кастомные special (если остались): зарегистрированы в plugins.specialScenes или через register() (§40).
  15. У ключевых врагов заполнен loot; победа показывает окно добычи.
  16. Проклятые предметы настроены в редакторе (эффект + сцена снятия).
  17. Проверить проект — без критических ошибок.
  18. Полное прохождение в index.html с нуля.
  19. Экспорт игры протестирован в чистой папке.

40. Plugin API для special-сцен (legacy и расширения)

Для лавок, лечения и покупок предпочтительны компоненты сцен и цепочки действий (§29) — без JavaScript. Раздел ниже — для сюжетных мини-игр и старых сцен с полем special, а также когда нужна уникальная логика в коде.

Движок поддерживает расширение без правки ядра двумя способами: JSON (plugins.specialScenes) или внешний JavaScript (SpecialSceneRegistry.register). Альтернатива без special: цепочка с say, show_choices, action: "chain:..." на выборе.

40.1. JSON-плагины (без кода)

Для простых сцен с текстом, выборами и флагами используйте секцию plugins.specialScenes в game_data.json:

{
  "plugins": {
    "specialScenes": {
      "dice_game": {
        "label": "Мини-игра в кости",
        "text": "Трактирщик предлагает сыграть. Ставка — 10 золотых.",
        "dialogue": [
          { "speaker": "Трактирщик", "text": "Ну что, рискнёшь?" }
        ],
        "choices": [
          { "text": "🎲 Играть (10 зм)", "to": "dice_roll", "showIf": { "goldMin": 10 } },
          { "text": "← Отказаться", "to": "tavern" }
        ]
      }
    }
  }
}

JSON-плагины автоматически регистрируются при загрузке данных. ID плагина должен совпадать с полем special у сцены.

40.2. JS-плагины (расширенная логика)

Для сложной логики (расчёты, циклы, обращение к состоянию игры) создайте отдельный JS-файл и подключите его после special-scenes.js:

<script src="js/special-scenes.js"></script>
<script src="plugins/my-custom-plugin.js"></script>

Регистрация обработчика:

SpecialSceneRegistry.register('alchemy_station', 'Алхимический станок', function(engine, sceneId, scene) {
  // Доступ к состоянию игры
  const gold = engine.state.gold;
  const hasHerb = engine.state.inventory.includes('herb');

  // Установка текста и диалогов
  engine.setText('Старый стол с колбами. ' + (hasHerb ? 'У вас есть ингредиенты.' : 'Нужны травы.'));
  engine.setDialogue([
    { speaker: 'Алхимик', text: 'Смешать зелье лечения? 5 золотых.' }
  ]);

  // Формирование выборов динамически
  const choices = [];
  if (hasHerb && gold >= 5) {
    choices.push({
      text: '🧪 Смешать зелье (5 зм)',
      action: 'alchemy_craft'  // Можно использовать action для вызова других методов
    });
  }
  choices.push({ text: '← Уйти', to: 'village' });
  engine.setChoices(choices);
});

41. Создание собственных систем правил

Движок поддерживает несколько RPG-систем через абстракцию RuleSystem. Вы можете создать собственную систему (например, упрощённую d6, FATE, GURPS) без правки ядра.

41.1. Наследование от RuleSystem

Создайте файл js/systems/my-system.js:

class MyD6System extends RuleSystem {
  get id() { return 'my_d6'; }
  get label() { return 'Моя d6-система'; }
  get description() { return 'Упрощённая система на кубиках d6.'; }

  // Переопределяем модификатор: d6 не использует (score-10)/2
  getModifier(score) {
    return Math.floor((Number(score) - 3) / 3); // Пример: 3-5=0, 6-8=1, 9+=2
  }

  // Point Buy или другая генерация характеристик
  getPointBuyConfig() {
    return { total: 12, min: 1, max: 6 }; // Распредели 12 очков, значения 1-6
  }

  // Расчёт ОЗ
  calculateHP(classKey, level, stats, data, conMod) {
    const base = data?.classes?.[classKey]?.hp || 6;
    return base + (conMod || 0) * Math.max(1, level);
  }

  // Расчёт защиты
  calculateAC(stats, equipment, data, engine) {
    const dexMod = this.getModifier(stats?.dex ?? 3);
    return 3 + dexMod; // Базовая защита 3 + мод. ЛОВ
  }

  // Бросок атаки
  rollAttack(attacker, target, engine) {
    const roll = engine.d(6);
    const total = roll + (attacker.atkBonus || 0);
    const ac = target?.ac || 3;
    const hit = total >= ac;
    const crit = roll === 6;
    let dmg = 0;
    if (hit) {
      dmg = engine.parseRoll(attacker.dmgRoll || '1d6') + (attacker.dmgBonus || 0);
      if (crit) dmg *= 2;
    }
    return { roll, total, hit, crit, dmg };
  }
}

const MySystem = new MyD6System();
if (typeof window !== 'undefined') window.MyD6System = MySystem;

41.2. Регистрация системы

Подключите файл после registry.js и зарегистрируйте:

<script src="js/systems/base-system.js"></script>
<script src="js/systems/dnd5e.js"></script>
<script src="js/systems/pathfinder2e.js"></script>
<script src="js/systems/my-system.js"></script>
<script src="js/systems/registry.js"></script>
<script>
  if (typeof SystemRegistry !== 'undefined' && typeof MyD6System !== 'undefined') {
    SystemRegistry.register(MyD6System);
  }
</script>

После регистрации система появится в выпадающем списке «Система правил» при создании нового проекта в редакторе.
